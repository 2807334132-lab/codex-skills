# 知识发现引擎 — /competitor init 规程

本模块定义 `/competitor init <dir>` 命令执行时使用的知识发现流程。
从指定目录的各类文件（MD/PDF/DOCX/TXT/HTML）中自动提取竞品实体、建立基线。

---

## Phase 1: 目录扫描

### 扫描范围
递归扫描目录下所有文件，重点关注格式：

| 扩展名 | 处理方式 | 注意 |
|--------|---------|------|
| `.md` | 直接读取 | Markdown 文件优先 |
| `.pdf` | 调 doc-parse 转 Markdown | 需 MinerU 可用 |
| `.docx` | 调 doc-parse 转 Markdown | 同上 |
| `.txt` | 直接读取 | 普通文本 |
| `.html` / `.htm` | 提取正文文本 | 去掉 CSS/JS/标签 |
| `.csv` / `.xlsx` | 尝试读取表结构 | 需 Bash + python 处理 |
| 其他 | 跳过 | 扫码/非文本格式跳过 |

### 排除规则
- `node_modules/`、`.git/`、`dist/`、`build/`、`__pycache__/`
- 二进制文件（.exe/.zip/.tar/.jpg/.png 等）
- 隐藏文件（`.` 开头的文件/目录）

### 扫描输出
```json
{
  "total_files": 15,
  "by_type": {"md": 5, "pdf": 3, "docx": 2, "txt": 3, "html": 2},
  "total_size_kb": 4520,
  "excluded": 8,
  "files": [
    {"path": "竞品调研/GPUStack分析.md", "type": "md", "size_kb": 120},
    ...
  ]
}
```

## Phase 2: 内容解析

### 并行处理池（≤5 并发）

```
对所有文件按类型分配处理管道 →
```

| 文件类型 | 处理管道 | 产出 |
|---------|---------|------|
| `.md` | 直接 Read | 原始文本 |
| `.pdf` | doc-parse → Read 输出 .md | 结构化 Markdown |
| `.docx` | doc-parse | 结构化 Markdown |
| `.txt` | 直接 Read | 原始文本 |
| `.html` | Bash: htmlq/pandoc 提取正文 | 纯文本 |

### 暂存格式

所有解析结果统一暂存到临时目录，格式：
```json
{
  "source_file": "竞品调研/GPUStack分析.md",
  "source_type": "md",
  "content": "# GPUStack ... \n\nGPUStack 是一个...",
  "meta": {
    "size_kb": 120,
    "processed_at": "2026-06-22T10:30:00",
    "parse_method": "direct_read"
  }
}
```

## Phase 3: AI 识别 + 差异分析

### 竞品实体提取

对汇总后的所有文本内容进行 AI 分析，提取：

| 识别维度 | 典型提取对象 | 示例 |
|---------|-------------|------|
| **竞品名称** | 公司名、产品名、平台名 | GPUStack、天工SkyClaw、MiniMax |
| **产品/技术描述** | 核心能力、技术栈、架构特点 | GPU Pooling、Multi-LoRA、Agent编排 |
| **客户/合作伙伴** | 客户案例、合作方、集成方 | 某银行、华为、中国电信 |
| **融资/估值** | 轮次、金额、估值、投资人 | A轮5000万/红杉 |
| **市场定位** | 品牌叙事、目标市场、Value Prop | "Token工厂"、"一人公司操作系统" |
| **威胁/机会信号** | 对昆仑元的直接/间接影响 | 叙事竞争、定价竞争、技术替代 |
| **行业背景** | 行业趋势、市场数据、报告引用 | "6层竞品矩阵"、"L1-L6" |

### 差异分析

识别出的竞品实体与已有基线对比：

```json
{
  "new_discoveries": [
    {"name": "MiniMax", "confidence": "high", "source": "竞品调研/Agent平台分析.pdf"},
    {"name": "智谱Agent", "confidence": "medium", "source": "竞品调研/智谱分析.md"}
  ],
  "fuzzy_matches": [
    {"name": "天工", "suggested": "昆仑万维天工", "source": "行业报告.html"}
  ],
  "already_tracked": [
    {"name": "GPUStack", "baseline": "GPUStack_跟踪日志_20260622.md"},
    {"name": "天工", "baseline": "天工_跟踪日志_20260622.md"}
  ]
}
```

### 分类逻辑

| 分类 | 条件 | 颜色标记 | 后续处理 |
|------|------|---------|---------|
| ✅ 新发现 | 名称清晰，有足够信息建立基线 | 绿 | 可导入 |
| 🔍 模糊匹配 | 名称模糊/信息不足/可能有歧义 | 黄 | 需要用户确认 |
| ⏭ 已在跟踪 | 名称与已有基线匹配（模糊匹配通过） | 灰 | 自动跳过 |
| 💡 低置信度 | 仅提到一次，缺乏上下文 | 蓝 | 仅记录不导入 |

```
✅ 新发现（可导入）━━━━━━━━━━━━━━━━━━━━━━━

    来源：竞品调研/Agent平台分析.pdf
    置信度：高
    → 输入 1 导入 | 输入 s 跳过

 2. 钉钉Agent      — 阿里旗下智能体平台
    来源：行业报告/钉钉分析.md
    置信度：高
    → 输入 2 导入 | 输入 s 跳过
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🔍 名称待确认（需人工核实）━━━━━━━━━━━━━━━━━━
 3. 天工 → 昆仑万维天工？
    来源：行业报告.html
    置信度：中
    → 输入 3 导入 | 输入 r 重命名 | 输入 s 跳过
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⏭ 已在跟踪中（自动跳过）━━━━━━━━━━━━━━━━━━━

💡 低置信度（仅记录）━━━━━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

导入命令：输入竞品编号（逗号分隔），如 "1,2" 或 "all"
或输入对应动作（s/r <数字> <新名称>）:
```

### 基线创建格式

确认导入后自动生成跟踪日志，格式：

```markdown
# {竞品名称} — 跟踪日志
> 基线建立日期：{YYYY-MM-DD}
> 来源：{init 时的来源文件路径}
> 发现方式：/competitor init — 自动发现
> 初始优先级：P3 背景（待评分）
> 状态：初版基线，信息待补充

## 产品概览
{从文件中提取的产品描述摘要}
{置信度标注：高/中/低}

## 已知信息
- {从文件中提取的结构化要点}
- {每个要点带来源文件引用}

## 待补充信息
- [ ] 官网/产品文档
- [ ] GitHub 统计
- [ ] 定价信息
- [ ] 融资信息
- [ ] 最新动态

## 来源文件
- {来源文件路径 1}
- {来源文件路径 2}

---

### 跟踪日志条目

#### {YYYY-MM-DD} — 初始基线（自动发现）
- 发现方式：`/competitor init` 从 {目录} 自动发现
- 关键发现：{从文件中提取的核心信息摘要}
- 待跟进：{需要人为确认/补充的内容}
```

---

## 输出文件

最终产出三个文件：

| 文件 | 路径 | 说明 |
|------|------|------|
| 发现报告 | `{竞品跟踪目录}/竞品发现报告_{YYYYMMDD}.md` | 本次发现的完整记录 |
| 跟踪日志 | `{竞品跟踪目录}/{竞品名}_跟踪日志_{YYYYMMDD}.md` | 每个确认导入的竞品生成一个 |
| 来源引用 | 嵌入跟踪日志头部 | 基线中保留每个信息的来源路径 |
